# План глобального рефакторинга логики подсказок

## Цель

Сделать подсказки частью единой обучающей модели, а не локальной UI-надстройкой. После рефакторинга приложение должно:

- выдавать подсказки последовательно и предсказуемо;
- учитывать режим, сложность стиха, стадию изучения и реальный прогресс пользователя;
- корректно связывать подсказки с рейтингом, мастерством и переходом между режимами;
- не ломаться при смене режима внутри одного и того же стиха;
- быть готовым к будущим AI-подсказкам без повторной переделки архитектуры.

## Что нужно исправить обязательно

### 1. Ввести сущность попытки

Нужен отдельный client-side слой `TrainingAttempt`, а не просто `hintState` внутри UI.

Минимальная модель:

- `verseId`
- `modeId`
- `phase` (`learning` | `review`)
- `difficultyLevel`
- `attemptStatus` (`active` | `completed` | `surrendered`)
- `progressSnapshot`
- `usedHints`
- `hintEvents`
- `ratingPolicy`

Главный инвариант:

- подсказки живут в рамках попытки;
- новая попытка начинается при смене `modeId` или при перезапуске упражнения;
- состояние подсказок не переносится автоматически между режимами одного стиха.

### 2. Заменить `onProgressChange(number)` на типизированный прогресс

Сейчас один и тот же `number` в разных режимах означает разное:

- номер чанка;
- число скрытых слов;
- число первых букв;
- число введенных слов.

Для подсказок это слишком хрупко. Нужен `ExerciseProgressSnapshot`, например:

```ts
type ExerciseProgressSnapshot = {
  kind: 'chunks' | 'word-order' | 'word-order-hinted' | 'first-letters' | 'first-letters-hinted' | 'first-letters-typing' | 'full-recall' | 'voice-recall';
  expectedWordIndex: number | null;
  completedUnits: number;
  totalUnits: number;
  isCompleted: boolean;
};
```

Главный инвариант:

- `HintEngine` не читает сырой state режима;
- каждый режим сам адаптирует свой внутренний state к общему snapshot-формату.

### 3. Вынести политику подсказок в отдельный доменный слой

Нужен отдельный модуль, например `src/modules/training/hints/`.

Рекомендуемые функции:

- `getAvailableHintsForAttempt(attempt)`
- `requestHint(attempt, hintType)`
- `resolveHintRatingPolicy(attempt)`
- `resolveHintBudgetEffect(attempt, hintType)`
- `buildHintContent(attempt, hintType)`

Главный инвариант:

- UI не решает сам, сколько стоит подсказка;
- UI не решает сам, портит ли подсказка рейтинг;
- UI только отображает policy и результат.

### 4. Сделать сложность общей конфигурацией упражнений

Нужен единый конфиг, например `exerciseDifficultyConfig.ts`.

Он должен задавать по каждой паре `mode + difficulty`:

- лимит ошибок;
- процент открытых слов;
- количество первых слов в hint;
- порог успешности recall;
- при необходимости плотность выбора/скрытия.

Главный инвариант:

- в упражнениях не должно остаться локальных `const MAX_MISTAKES_BEFORE_RESET = 5`;
- reveal ratio и penalty rules не должны быть разбросаны по компонентам.

### 5. Разделить подсказки по силе воздействия

Нужна нормальная шкала помощи:

- `context`
- `firstWords`
- `nextWord`
- `targetedReveal`
- `surrender`

И отдельная policy для learning/review:

- в `learning` часть мягких подсказок может не ронять попытку в `0`, но должна ограничивать максимальный рейтинг;
- в `review` любая содержательная подсказка должна автоматически переводить попытку в `rating = 1` или максимум `1`;
- `surrender` всегда завершает попытку как `0`.

Главный инвариант:

- пользователь не должен вручную выбирать высокий рейтинг после реальной подсказки, если policy это запрещает.

### 6. Подготовить AI-слой как надстройку, а не основу

AI нужен только после стабилизации детерминированной модели.

AI можно подключать к:

- генерации семантической подсказки;
- контекстной подсказке по смыслу стиха;
- адаптивному выбору типа следующей подсказки;
- пост-анализу частых ошибок пользователя.

AI не должен:

- сам менять мастерство;
- сам выбирать рейтинг;
- обходить hint policy.

## Нужны ли изменения в Prisma / backend

### Сейчас

Большой обязательной миграции в `prisma/schema.prisma` не требуется. Текущая модель `UserVerse` достаточна для первого большого refactor, если держать attempt-state на клиенте.

### Позже, если нужен максимум пользы

Если ты хочешь:

- синхронизацию подсказок между устройствами;
- честный дневной лимит на сервере;
- аналитику по тому, какие подсказки реально помогают;
- персонализированную AI-адаптацию;

тогда лучше не раздувать `UserVerse` новыми колонками, а добавить отдельную событийную сущность, например:

- `TrainingAttemptEvent`
- или `HintUsageEvent`

Минимальный payload события:

- `telegramId`
- `externalVerseId`
- `modeId`
- `hintType`
- `phase`
- `difficultyLevel`
- `progressBefore`
- `createdAt`

Решение:

- для refactor v1 backend можно не трогать;
- для v2 лучше добавить event-table, а не хранить hint-history в `UserVerse`.

## Поэтапный план

### Фаза 1. Нормализовать модель

- Ввести `TrainingAttempt`.
- Ввести `ExerciseProgressSnapshot`.
- Ввести `HintPolicyResult`.
- Убрать прямую зависимость `useHintState` от локального UI-смысла.

### Фаза 2. Починить жизненный цикл

- Сбрасывать attempt-state при смене `modeId`, а не только стиха.
- Разделить `attempt complete` и `attempt surrendered`.
- Запретить перенос `surrendered` в следующий режим.

### Фаза 3. Собрать единый `HintEngine`

- Перенести `getAvailableHints`, `computeMaxRatingFromHints`, budget-логику и генерацию контента в единый модуль.
- Сделать один вход: `requestHint(attempt, hintType)`.
- Сделать один выход: обновленный `attempt + hint result`.

### Фаза 4. Собрать единый `ExerciseDifficultyConfig`

- Перенести туда ошибки, reveal ratios, recall thresholds и силу стартовых hints.
- Подключить все режимы к этому конфигу.
- Удалить локальные магические числа из режимов.

### Фаза 5. Привязать rating к фактической помощи

- Переписать `TrainingRatingButtons`, чтобы они всегда уважали `ratingPolicy`.
- В review не позволять `rating >= 2` после содержательной подсказки.
- В learning явно определить, какие hints режут максимум до `2`, `1` или `0`.

### Фаза 6. Добавить тестовый контур

- unit tests для `HintEngine`;
- unit tests для `ExerciseProgressSnapshot` adapters;
- unit tests для `ratingPolicy`;
- integration test: `surrender` в одном режиме не переносится на следующий;
- integration test: `nextWord` берет корректный индекс в hinted modes;
- integration test: rating buttons не дают выбрать запрещенный рейтинг.

### Фаза 7. Только после этого подключать AI

- AI-подсказка по смыслу;
- AI-объяснение ошибки;
- AI-выбор следующего мягкого hint;
- AI-адаптация по истории hint-events.

## Что делать в коде первым

Порядок внедрения:

1. Создать доменные типы `TrainingAttempt`, `ExerciseProgressSnapshot`, `HintPolicyResult`.
2. Переписать `TrainingSession`, чтобы hint-state сбрасывался по `verseId + modeId + attemptRevision`.
3. Переписать режимы на `reportProgress(snapshot)`.
4. Переписать `TrainingRatingButtons` на policy-driven API.
5. Вынести difficulty rules в общий конфиг.
6. Только потом переписать drawer и hint UX.

## Критерий завершения

Рефакторинг можно считать успешным, когда:

- подсказки больше не живут как локальный разрозненный state;
- рейтинг всегда вычисляется из policy, а не из честности пользователя;
- смена режима не наследует hint-state;
- сложность действительно влияет на каждый режим единообразно;
- AI можно подключить как отдельный поставщик hint-content, не трогая логику прогресса.
